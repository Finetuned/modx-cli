# Internationalization (i18n) & Localization (l10n)

MODX CLI outputs all command messages in the user's language. The translation system is built on the [Symfony Translation component](https://symfony.com/doc/current/translation.html) and driven by YAML files. Five locales are bundled out of the box, and adding a new one requires only creating a directory of YAML files.

| `es` | Spanish  | |
| `ru` | Russian  | |

## Locale Selection

The CLI resolves the locale in priority order. The first match wins:

1. **`--locale` CLI option** (highest priority)

   ```bash
   modx --locale=fr resource:list
   ```

2. **`MODX_CLI_LOCALE` environment variable**

   ```bash
   export MODX_CLI_LOCALE=de
   modx resource:list
   ```

3. **`LANG` or `LC_ALL` system environment variables**

   If your shell is already set to a supported locale, the CLI picks it up automatically:

   ```bash
   LANG=es_ES.UTF-8 modx resource:list
   ```

4. **PHP system locale** (`Locale::getDefault()`)

5. **Falls back to `en`** if none of the above yield a supported locale.

**Region codes are normalized automatically.** `fr_FR`, `fr_FR.UTF-8`, and `fr` are all treated as `fr`.

## Translation Domains

Strings are split across four domain files per locale. Each file lives at `translations/{locale}/{domain}.yaml`.

| Domain | Purpose | Example keys |
|--------|---------|--------------|
| `errors` | Shared error messages used across many commands | `operation_aborted`, `not_found` |
| `messages` | General informational messages | Status labels, prompts |
| `commands` | Per-command runtime strings (success, failure, confirm prompts) | `resource.create.success`, `config.add.added` |
| `validation` | Input validation error messages | Field-level errors |

## i18n Commands

All four i18n management commands run without a connected MODX instance (`MODX = false`).

### i18n:stats

Show a coverage table for all locales and domains.

```bash
modx i18n:stats
```

Filter to a single domain:

```bash
modx i18n:stats --domain=commands
```

JSON output (useful for scripts):

```bash
modx i18n:stats --json
```

Sample output:

```
Translation Coverage Statistics

 Locale  Domain      Total  Translated  Missing  Coverage
 de      commands    312    312         0        100.0%
 de      errors      28     28          0        100.0%
 fr      commands    312    312         0        100.0%
 ...
```

---

### i18n:validate

Check all translation files for completeness (missing keys) and empty values. Exits with code `1` if any issues are found — suitable for CI pipelines.

```bash
modx i18n:validate
```

Validate a specific locale only:

```bash
modx i18n:validate --locale=fr
```

JSON output:

```bash
modx i18n:validate --json
```

---

### i18n:missing

List all keys present in the English baseline (`translations/en/`) that are absent in a target locale. `locale` is a required positional argument.

```bash
modx i18n:missing fr
```

Limit to a single domain:

```bash
modx i18n:missing de --domain=commands
```

Exits with code `1` if any keys are missing.

---

### i18n:extract

Scan `src/` PHP files for `trans()` / `transChoice()` / `ErrorMessages::*` calls and compare the keys found in code against those in the YAML files.

```bash
modx i18n:extract
```

This reports:
- **New keys** — present in code but not in any YAML file
- **Orphaned keys** — present in YAML but not referenced in code

Write placeholder entries for new keys into the base `en/` YAML files:

```bash
modx i18n:extract --update
```

Limit to a single domain:

```bash
modx i18n:extract --domain=commands
```

Typical workflow after adding new `trans()` calls to a command:

```bash
# 1. Write empty placeholders to en/ YAML
modx i18n:extract --update

# 2. Fill in the English values in translations/en/
# 3. Run validate to confirm no issues
modx i18n:validate

# 4. Repeat for each non-English locale (see Contributing Translations)
```

## YAML File Format

Translation files use nested YAML with dot-notation keys that mirror the command namespace hierarchy.

**Key structure:** `{namespace}.{action}.{message_type}`

**`translations/en/commands.yaml`** (excerpt):

```yaml
resource:
  create:
    success: 'Resource created successfully'
  erase:
    not_in_trash: 'Resource "%pagetitle%" (ID: %id%) is not in the trash'
    irreversible_warning: 'WARNING: This operation is irreversible!'
config:
  add:
    exists_confirm: 'Instance "%name%" already exists. Do you want to overwrite it?'
    added: 'Instance "%name%" added'
```

**`translations/fr/commands.yaml`** (same keys, French values):

```yaml
resource:
  create:
    success: 'Ressource créée avec succès'
  erase:
    not_in_trash: 'La ressource "%pagetitle%" (ID : %id%) n''est pas dans la corbeille'
    irreversible_warning: 'ATTENTION : Cette opération est irréversible !'
config:
  add:
    exists_confirm: 'L''instance "%name%" existe déjà. Voulez-vous l''écraser ?'
    added: 'Instance "%name%" ajoutée'
```

### Parameter placeholders

Parameters use Symfony's **double-percent** syntax: `%param_name%`.

```yaml
# Correct
added: 'Instance "%name%" added'

# Wrong — curly braces are not supported
added: 'Instance "{name}" added'
```

At runtime, pass the value in the `$params` array:

```php
$this->trans('config.add.added', ['%name%' => $name], 'commands');
```

### Apostrophes in YAML strings

Single-quoted YAML strings escape apostrophes by doubling them:

```yaml
# Correct — double single-quote inside single-quoted string
failed: 'Échec de l''effacement de la ressource'

# Also correct — use double-quoted string instead
failed: "Échec de l'effacement de la ressource"
```

## Contributing Translations

### Adding missing strings to an existing locale

1. Find what's missing:

   ```bash
   modx i18n:missing fr
   ```

2. Open the relevant file (e.g. `translations/fr/commands.yaml`) and add the translations.

3. Validate:

   ```bash
   modx i18n:validate --locale=fr
   ```

4. Run the integration test suite to catch any placeholder inconsistencies:

   ```bash
   vendor/bin/phpunit --testsuite integration --no-coverage
   ```

### Adding a new locale

1. Create the locale directory and copy the English baseline:

   ```bash
   mkdir translations/ja
   cp translations/en/*.yaml translations/ja/
   ```

2. Translate all values in each of the four YAML files. **Keep the key structure identical to `en/`.**

3. Validate the new locale:

   ```bash
   modx i18n:validate --locale=ja
   ```

4. Run the integration tests (they will automatically pick up the new locale):

   ```bash
   vendor/bin/phpunit --testsuite integration --no-coverage
   ```

## CI Integration

Add these steps to your CI pipeline to catch translation regressions:

```bash
# Fail if any locale has missing or empty translation keys
modx i18n:validate

# Fail if French is missing any keys (run per locale as needed)
modx i18n:missing fr
modx i18n:missing de
modx i18n:missing es
modx i18n:missing ru

# Run integration tests — includes TranslationValidationTest which checks
# YAML syntax, key completeness, and %param% placeholder consistency
vendor/bin/phpunit --testsuite integration --no-coverage
```

## Troubleshooting

| Symptom | Likely cause | Fix |
|---------|-------------|-----|
| Output still in English despite `--locale=fr` | `--locale` must come before the command name | Use `modx --locale=fr resource:list` not `modx resource:list --locale=fr` |
| `TranslationValidationTest` fails with missing keys | A new `trans()` key was added to `en/` but not to other locales | Run `modx i18n:missing <locale>` for each locale and fill in the values |
| YAML parse error | Unescaped apostrophe in a single-quoted string | Double the apostrophe: `l''instance` or switch to double-quoted string |
| Locale not recognized / falls back to `en` | The locale directory does not exist under `translations/` | Create `translations/{code}/` with all four domain files |
| `i18n:missing` reports 0 missing but strings show the key name at runtime | Key exists in YAML but value is empty | Run `modx i18n:validate --locale=<code>` to find empty values |
